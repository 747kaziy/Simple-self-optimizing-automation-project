## 1. 基础概念与目标

在看到这个 Github 仓库时，我大概花了一天左右时间了解它。在此之前我已经用过市面上大多数 AI 辅助工具：ClaudeCode、Cursor、Trae，以及各种主流模型。

- 用 IDE 编辑器时，它会自动帮我补全，我全程只需要 Review 和 Tab，这些 IDE 也知道设置对应的 rules 规范输出。
- 使用网页版（如 DeepSeek）时，最常用的是上传文件和 CV（复制粘贴）。

但我很少使用过类似 openclaw 这种通读项目结构的成熟 Agent。在阅读完文章后，我意识到这完全可以理解为一个**自动化辅助工作流框架**。我开始着手模仿并定制一个符合自己日常开发的个性化框架。

🎯 **最终目标：轻量级、全面、质量和效率并齐。**

文章开头，我先定义一个核心公式：

> **工作效率 = 底层大语言模型能力 × 生成器输出能力**

各大厂商在卷前面的乘数，而咱们通过架构设计拉升后面的乘数。以下所有的折腾，都是以提升“生成器输出能力”为标准。

在此之前，首先需要用大白话对齐三个关键术语：

- **Prompt（提示词）**：给 A 立的规矩和下的指令。分两种：一种是“系统提示”（设定人设和边界），另一种是“单元任务提示”（将复杂功能拆成小指令，降低出错率）。
- **Skill（技能库）**：你可以把它理解成一个“问题模板库”。比如你经常问 JVM 调优，就把最佳问法、必填参数整理成一个 .md 模板。下次直接套用，相当于给 A 装了个“专家插件”。
- **Context / Memory（上下文与记忆）**：为了解决 A“一换聊天窗口就失忆”的毛病。把关键信息（你的名字、公司技术栈）存下来，新开窗口自动带上，不用每次从头交代。

## 2. 核心方法论

接下来是这个仓库的两篇文章，也是“胶水编程”和《A Formalization of Recursive Self-Optimizing Generative Systems》的精髓。

- **胶水编程**：组合 > 创造。找网上的轮子，不做重复劳动，只做粘合剂。
- **递归自优化（四个核心思想提炼）**：

> 1. **优化目标从“产物”转移到“生成器”**
>    - **痛点**：当 A 写出 Bug，K 习惯直接说“这里报错了，改成这样”，这只是治标不治本。
>    - **落地**：不要只修 Bug，要修 Prompt/Context。发现错误后，强制 A 提取经验生成规则写入 .traerules，确保绝不再犯。
> 2. **物理隔离生成器与优化器**
>    - **痛点**：同个对话框让 A“写代码 → 检查自己代码”，极易产生“幻觉闭环”。
>    - **落地**：左右互搏。切断自我审查，开个新窗口用模型 B 当优化器去挑刺，再反馈给 A。
> 3. **实施递归闭环自优化工作流**
>    - **落地**：A 犯错 = 生成器需要校准。要求 A 识别“上下文盲区”并更新规则。这就是“人工梯度下降”，把 K 的习惯注入 A 的推理路径。
> 4. **追求不动点语义，打造稳定的生成器**
>    - **目标**：Vibecoding 的最高境界。当规则、文档、记忆完美适配当前业务时，K 只需输入极简意图（“加个登出按钮”），A 就能输出完美符合规范的代码。

## 3. 思路演化

理论存在，实践开始。

### V1.0 面向模板：解决“有没有标准”的问题

第一步，设计一个简易的框架目录：

```
personal-framework/
├── .traerules       # 规则
├── .vibe/           # 底层逻辑
│   ├── logic-core.md    # 静默递归闭环
│   └── logic-evolve.md  # 系统自举进化
├── MEMORY.md        # 上下文记忆库
└── skills/          # 技能库
```

- **第二步：搭建上下文记忆库。** 核心是追溯（变化日志）和短期记忆。新窗口优先读它。痛点是文件会变冗长，需要 A 自己去维护精简，千万不能过多人工干预。
- **第三步：构建初始技能库。** 我搞了个“元 Skill”（生产 Skill 的 Skill），以及“审核器”和“优化器”Skill，专门用来自我迭代（需物理隔离）。
- **第四步：提示词规范。** 喂给 A 一堆原则，生成一套完美的提示词模板。

**🚫 V1.0 痛点**：繁琐！每次要像做英语完形填空一样去填模板。遇到不熟悉的领域连怎么填都不知道。填空的人不同，输出依然有偏差。

### V1.1 面向指令：将工作流封装成 API

既然完形填空累，为什么不把活儿丢给 A？把工作流封装成带标识符的指令（类似 API）。

- 定义 /prompt：K 输入 /prompt (我要做个网站)，A 会主动交互确认，最终反吐出标准 Prompt。
- 定义 /skill：K 输入 /skill (websiteSkill, 精通电商网站)，A 自动生成专家级 .md 文件。
- 之后只需输入 /prompt ("加个购物车", /websiteSkill)，A 就秒懂调用哪个专家。总结记忆同理，用 /cleanMemory 搞定。

**🚫 V1.1 痛点**：

1. A 会犯蠢，不认指令当普通聊天（解法：加持强约束）。
2. K 得记指令，中断心流（解法：加 /help 检索）。
3. 审核器和生产器如果物理隔离在两个窗口，数据怎么共享？（解法：文件系统共享）。

### V1.2 面向意图与混合模式

基于以上痛点，我设想过全自动的 V1.2 版本：让 A 自己猜 K 的意图（比如 A 主动提议建 Skill 或改规则）。
但纯意图驱动太不可控，容易引发幻觉暴走。所以我最终倾向于 **指令 + 意图混合模式**。A 可以主动感知并提议，但最终执行的“扳机”必须交还给 K（比如回复 Y 确认）。

## 4. V2.0：自动化工程专家 

本来以为写到 V1.2 就玩明白了。但翻了翻最近社区大佬们的硬核玩法和顶级论文，我的格局还是小了！我趁热打铁，把接下来要优化的 V2.0 降维打击思路记录下来：

### 思考一：提示词工程的尽头是上下文工程

其实思路一直在被提示词工程带弯了，之前一直在思考如何写出标准的提示词或者让A帮你写出标准的提示词，其实应该转为维护上下文工程。

V1.1 还在纠结怎么写提示词，这思路确实落后了。在某个文章看到的PRP框架，简单理解就是
以后K不再写保姆级 Prompt 了。K 只负责写大白话业务需求 PRD.md。

- **P (PRD - 业务需求)**：大白话需求（“加个短信登录页”）。
- **R (Rule - 技术规范)**：预设的 websiteSkill.mdc（“用vue，禁用React”）。
- **P (Project Context - 项目上下文)**：IDE 扫描当前项目现状。

只需敲下一行极简指令：/build docs/PRD.md  /websiteSkill
此时，**A 不再是代码生成器，而是context（上下文）编译器**。不要试图用花言巧语控制 AI，让 AI 自己去读结构化的文件。

### 思考二：拆分规则成多个文件规则

其实这个很好理解，体现在数据维护上就是分库分表，体现在项目上就是模块化，微服务思想。

- **【过去】**：把所有前后端、数据库规范全塞进一本 500 页的《员工手册》（巨大的 .traerules 里）。A 的注意力被严重稀释，经常“看了后面忘前面”。

- **【V2.0】**：大锅饭砸碎！给 A 配备对于的子文件夹，如果是x相关的就查x相关的规则（.vibe/rules/ 下的独立小文件如 db.mdc, react.mdc）。

  这里解释一下mdc这个文件格式，它在 Markdown 的基础上，增加了“元数据 ”和“作用域 ”，专门用来告诉 AI：*“你应该在什么时候使用我，以及你应该如何遵守这些规则。”*
  利用 MDC 路由，在文件顶部写触发条件。当你让 A 写前端样式时，它绝对不会加载数据库规则，此时的 A 是一个 100% 专注的页面仔，零杂念零幻觉！

### 思考三：记忆库拆分

- **【过去】**：一个又长又臭的单体备忘录 MEMORY.md。
- **【V2.0】**：像人类大脑一样的“三级记忆系统”：
  1. **长记忆 (MEMORY_CORE.md)**：只存项目命脉和血泪教训（如“鉴权强依赖 Token 绝对禁改”），极度精简，永远挂载。
  2. **动态计划 (TODO.md)**：状态机便利贴，拆解任务，做完打 [x]。
  3. **短记忆与状态压缩 (PROGRESS.md)**：神级操作！跟 A 聊了 30 轮，A 开始幻觉胡言乱语。输入 /compress这个定义的总结指令。A 会立刻停下，把有效信息浓缩成 200 字快照写入文件。然后你就可以开新的窗口。

### 思考四：强制前置规划与 隔离测试

原理很简单，先拆分任务再做作业，遇到复杂问题先拆成单元问题测试解决。

- **强制前置规划 (/plan 指令)**：在系统规则里死死按住 A 的键盘：“不管需求多小，禁止直接写代码！”A 必须先输出修改计划，只有 K 回复 **Y**，A 的键盘才解锁（Human-in-the-loop，绝对的安全感）。
- **隔离测试(/ladder指令)**：当 A 在 1000 行文件里修 Bug 陷入死循环时，输入此指令。A 会立刻停手！新建空白 test.js（降维到玩具级环境），跑通核心逻辑后，再反推合入复杂文件中（隔离测试）。

### 思考五：MCP 工具集

以前靠纯文本传递信息（比如去 Navicat 复制表结构给 AI），太 Low。现在的终极形态是 **MCP**插件体系。给 AI 装上“手和眼睛”，让它自己在后台执行 SQL 查表、查官方文档。

结合之前的元指令思想，我搞了一个**终极指令**：/buildMCP [需求]。
比如输入 /buildMCP (我要一个读取本地 Redis 缓存的插件)。A 会自己去写出 Node.js 代码，自己挂载到 Trae 设置里，最后**调用自己刚写好的工具去帮我查 Redis**！这才是真正的递归自我进化！

写到这里，这里其实有个很大的问题，那就是造物主悖论问题

**如果用来让 A 生成 MCP 的那个“元指令（Meta-MCP）”本身就是个半吊子怎么办？** 如果它写出来的插件全报错、跑不起来，那“自己给自己造工具”岂不是成了一个无限报错的死循环？

首先，在 V2.0 下绝对不能去手动修那个烂插件，而是要**对“元 MCP”本身进行工程化降维打击**。我构思了 4 个维度的手段来优化它：

1. 第一步，喂官方知识
   AI 为什么会写错代码？因为它是靠以前学过的杂乱知识在“脑补”。比如它要造个新插件，它可能会用几年前的老语法。我的办法很简单：直接去官网把最新的 API 文档下载下来，存成一个文本文件。然后在母机的规则里死死规定：“你要造工具，必须且只能看着这本官方说明书写，绝不允许凭空捏造。”这就从源头上掐断了它胡编乱造的可能。
2. 第二步，提供一个标准模板
   AI 最强大的能力是“照猫画虎”。哪怕有说明书，让它从零建文件夹、写配置文件也很容易出错。所以，我找了一个已经完美跑通的插件，把里面的业务逻辑掏空，只留下最标准的目录结构和日志打印格式，当成一个“黄金空壳模板”。我告诉它：“以后你每次造新工具，直接拿这个模具去复刻，你只管往里面填核心逻辑就行。”这样一来，出错的概率直线下降。
3. 第三步，先测试再生产
   造出来的工具好不好使，不能拿我的编辑器当小白鼠。我定了个死规矩：A 写完新插件的代码后，绝对禁止直接挂载到系统里！它必须先在本地建一个独立的测试脚本（比如 test.js）跑一下。这就好比造好发动机先在测试台上轰几脚油门。如果报错了，A 必须自己把报错日志看一遍，自己去改；只有终端显示“测试通过”，它才有资格把这个新插件装到我的系统上。这就把危险完全隔离在了沙箱里。
4. 第四步，保证同一个坑绝不踩两次。
   有时候，母机造出来的工具能跑通，但存在逻辑硬伤。比如我发现它造的 Redis 插件，一遇到网络波动就死循环卡死。这说明母机脑子里根本没有“断线重连”这个概念。这时候，我绝对不去改那个 Redis 插件的代码，我直接在聊天框敲一句指令：/evolve (我发现你造的插件一断网就死循环，你赶紧去反思一下)。神奇的事情就会发生：A 会自动打开它自己的底层核心规则文件，把“造任何工具都必须加上错误监听和重连逻辑”这条规矩，永远地刻进自己的脑子里。这就相当于我给它做了一次基因突变。下次它再造工具，绝对自带防断网功能。

只要把这 4 步跑通，随着时间的推移，你的“元 MCP”就会进化成一个**百发百中、量产神器的终极母机**！

## 5.V2.0 是否摒弃前面的核心思想了吗？

写到这里，我猛地停下来反思了一下：我搞了这么多 MDC、MCP、状态机拆分，把系统折腾得这么复杂，是不是自己优化嗨了，把最开始那四个“理论基石”给忘在脑后了？

如果说 V1.0 和 V1.1 只是在生搬硬套地练习理论招式（比如死磕提示词、手写那本厚得要命的规则手册），那 V2.0 就是彻底打通了任督二脉。它不是在做加法，而是在做“自动化”，把那些虚无缥缈的理论变成了系统底层的齿轮。

你看看这四条，哪一条不是 V2.0 的写照：

第一，**“优化生成器”**。以前出了 Bug 我就盯着代码修，那叫治标。现在 Bug 出来了，我根本不碰代码，直接去改 PRD 或者对应的技能文件（比如 react.mdc）。配合 MDC 动态路由，这就叫“靶向优化”，我改的是生成器的“出厂参数”，这才是真正的优化生成器。

第二，**“物理隔离”**。以前我让 AI 自己写代码又自己查，它肯定觉得“我写得完美无缺”。现在我搞了这一套，要写代码就去当前窗口，要重置状态就用 /compress 存个档开个新窗（时间隔离），要修疑难杂症就抽离到空白的 test.js 去测（空间隔离）。这本质上就是把 AI 的“作案现场”和“审判法庭”彻底分开了，幻觉闭环被掐得死死的。

第三，**“闭环自优化”**。这块最明显，你看 A 自己造 MCP 工具，发现工具不好使我就用 /evolve 指令去骂它，它自己再去改自己的核心规则文件。这整个过程，不就是它在一步步自我生长、自我迭代吗？这完全符合那种“生成器参数进化”的路径。

第四，**“稳定生成”**。你看我现在干活，压根就不跟它多废话。我敲下 /build 这个指令，它自己去读 PRD、去读规范、去读上下文。整个过程没有任何人的情绪干扰，也没有那些花里胡哨的提示词，直接输出符合预期的代码。这不就是咱们梦寐以求的那个“不动点语义”吗？

所以说，V2.0 压根就没摒弃它们。它只是把当初那些写在纸上的理论，变成了这套自动化框架里的“骨架”和“血液”。只要你用上了这套流程，你随手敲下的每一个指令，其实都在不知不觉中运行在这四个思想的最高轨道上

## 6. 总结

终极一句话串联所有 V2.0 思想：

> 你写了一份大白话需求（**PRP 框架**）；敲下 /plan，A 瞬间挂载对应规范（**MDC 路由**）；A 写出计划，你点头确认（**强制前置规划**）；A 带着工具去查表写代码（**MCP 工具**）；工具不够用且通过了沙箱测试，A 写个脚本给自己扩充功能（**自举进化**）；卡住了自动新建测试（**LADDER 降维**）；干完活把坑写进长期记忆并存档当前进度（**三态记忆压缩**）...

在这个流转中，你不再是一个在聊天框里跟 AI 纠结代码怎么写的人。**你是一个高高在上的技术总监 (K)，而 Trae + V2.0 框架，就是你手下那个永远不知疲倦、极其守规矩、还会自己写工具给自己用的高级研发团队 (A)。**
